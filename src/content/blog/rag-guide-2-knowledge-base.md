由RAG技术详解1可知，RAG技术使用最典型的地方就是企业级知识库，所以知识库的构建可谓相当重要，而知识库构建的前提是要有知识，所以第一步就是文档收集，我们要保证文档的质量，其次就是数据清洗，把无关的东西过滤以减少噪声，然后就是文档切分了，你需要把文档切成一片一片，否则和直接扔给AI就没区别了，而切分后的片如何可以和用户的问题匹配，就需要embedding后根据向量相似度来匹配。所以构建知识库的流程就是 **文档收集->文档清洗->文档切分->切片向量化->存入向量数据库**。

![Pasted image 20260608225909](/blog-assets/rag-guide-2-knowledge-base/Pasted%20image%2020260608225909.png)

我们来自顶向下地看一下每一步的目的。
![Pasted image 20260608230414](/blog-assets/rag-guide-2-knowledge-base/Pasted%20image%2020260608230414.png)

**文档收集->文档清洗->文档切分->切片向量化->存入向量数据库**，这条链路的目的就是提供**完美的知识库**供后续使用。

**文档收集->文档清洗->文档切分->切片向量化**，这条链路的目的应该看上一条链路的需求，即需要完美的数据来存入向量数据库，所以这一步的目的是要得到**完美的数据**。

**文档收集->文档清洗->文档切分，** 依旧回看上一条链路的需求，完美的数据，而这个数据的由来便是把切片向量化，所以很明显，这一步的目的是得到**完美的切片**。

**文档收集->文档清洗，** 由上一步的到，我们需要完美的切片，而切片是从文档内容切分而来，我们需要**高质量的内容**，所以文档的质量很关键，这两步其实可以看为一个阶段的两个部分，首先我们需要收集到高质量的文档，否在后面即使玩出花，也是“shit上雕花”，所以高质量的文档是基础，但是文档有不同的格式，不同文档也可能有重复的部分，这些和我们需要的内容无关的内容，我们统称为“噪声”，它会污染整个知识库，我们需要明确，**“噪声”是不可避免的，但是我们需要尽可能的减少“噪声”**。而文档结构，我们也需要稍微转化一下，让其适合切分。
![Pasted image 20260608231611](/blog-assets/rag-guide-2-knowledge-base/Pasted%20image%2020260608231611.png)


上述自顶向下概述了一遍知识库构建的整个链路，看起来简单，但其实每一步都有一些门道，我个人认为，每一步都很重要，都会很大的影响知识库的质量，但最重要的还是文档切分，因为尽管前两者很重要，但技术难度却很低，你只需要找到高质量的文档即可，而文档切分却有很多种不同的切法，根据不同的场景或者不同的资料，你都要选用不同的切法，就很麻烦。而切好的切片是要向量化放进向量数据库的，所以你切的东西其实就是你后面要用的东西。
接下来我会详细的讲解每一步需要注意哪些，以及如何做的更好。
## 文档收集

文档收集是 RAG 知识库构建的第一步，重点不是“把所有文档都丢进去”，而是要判断哪些资料值得进入知识库。
![Pasted image 20260608231629](/blog-assets/rag-guide-2-knowledge-base/Pasted%20image%2020260608231629.png)
需要关注以下几点
- **来源权威**：优先选择官方文档、内部制度、产品说明、技术规范、FAQ、API 文档等可信资料。如果资料来源不可靠，后续生成的答案也可能不可靠。
- **内容完整**：文档应尽量覆盖用户可能提问的主要问题。如果知识库缺少关键内容，RAG 检索不到相关资料，模型就容易回答不完整，甚至产生幻觉。
- **内容最新**：RAG 的优势之一是可以接入最新资料，因此要注意文档版本和更新时间。过期文档可能会导致系统给出错误答案。
-  **避免内容重复**：同一份资料可能在多个地方重复出现，例如官网文档、内部 Wiki、PDF 手册中内容相似。重复内容过多会影响检索效果，也会增加存储和处理成本。
- **权限控制**：企业知识库中可能包含敏感信息，例如合同、薪资、客户资料、内部策略等。 文档收集时就需要考虑哪些内容可以被哪些用户访问。
- **文档质量**：有些文档虽然存在，但结构混乱、内容碎片化严重，或者只是临时记录，并不适合直接进入 RAG 知识库。这类内容需要先整理后再接入。
简单来说，文档收集阶段的核心目标是：

> 找到权威、完整、最新、可用、可控的资料，为后续检索和生成打好基础。

## 文档清洗
文档清洗是指对收集到的原始文档进行整理、过滤和规范化处理，让文档变得更适合后续的切分、向量化和检索。
![Pasted image 20260608231718](/blog-assets/rag-guide-2-knowledge-base/Pasted%20image%2020260608231718.png)
首先我们要明白文档清洗的目的是什么，其实目的就一个，让文档内容更适合后续切分、向量化和检索。更明确一点就是：**去除噪声、保留有效信息、规范文档结构**。

原始文档往往不是直接可用的，里面可能包含很多与正文无关的内容，例如：

- 页眉页脚
- 目录
- 广告信息
- 导航栏
- 版权声明
- 重复段落
- 多余空行
- 乱码
- HTML 标签
- 格式混乱的表格
- 与正文无关的图片说明

如果这些噪声不处理，它们也会被向量化并进入知识库，导致检索结果不准确，甚至让模型生成错误答案。

清洗文档首先要提取正文内容，减少噪声干扰。但对于不同来源的文档，提取方式不一样。
![Pasted image 20260608231743](/blog-assets/rag-guide-2-knowledge-base/Pasted%20image%2020260608231743.png)

去除无关噪声

对于重复内容、多余空行、乱码等，这些完全无意义的内容，都是噪声，我们应该去除这些无关的噪声。

保留有用结构
- 清洗不是简单删除内容，还要尽量保留有助于理解的结构，例如：
    
	- 标题层级
    
	- 段落关系
    
	- 列表
    
	- 表格字段
    
	- 代码块
    
	- 参数说明
    
	- 示例内容

- **规范格式**
    
	- 将混乱的格式整理成统一、清晰的文本格式，方便后续文档切分。例如统一空行、标题格式、表格格式等。


简单来说，文档清洗的核心目标是：

> 让进入知识库的内容尽可能干净、准确、结构清晰，减少噪声对检索和生成结果的干扰。
## 文档切分（个人认为最重要）

文档切分很容易理解，就把一整篇文档，按照一定规则拆成多个较小的“文本片段”，这些片段通常叫做 **chunk** 或 **切片**。这些切片将会是未来检索时候的直接资料。

为什么要切分？因为寻找资料的时候你不可能直接把所有资料全部给AI，那样就失去了RAG的意义，所以需要切分，这样检索的时候就可以按需提取，只找到相关的那一段内容。
![Pasted image 20260608231846](/blog-assets/rag-guide-2-knowledge-base/Pasted%20image%2020260608231846.png)
主要目的：
- **降低文本长度**：长文档太大，不适合直接向量化，也不适合直接放进模型上下文。
- **提高检索精度**：切得合适，用户问题更容易匹配到相关内容。
- **保留语义完整性**：每个切片最好表达一个相对完整的意思，不能把一句话或一个步骤硬切断。
- **减少无关信息干扰**：只召回相关切片，而不是整篇文档，可以减少噪声。
- **方便后续生成回答**：模型拿到更精准的上下文，回答质量会更高。

切分有两个重要的点，一个是**颗粒度**，一个是**切分策略**。

**颗粒度**指每个文本块有多大。它决定了一个 chunk 里包含多少信息。颗粒度小，检索更精准，适合定位具体事实；颗粒度大，语义更完整，适合保留上下文。

首先理解两个名词。
Chunk：切分后的文本块，是检索和送入模型的基本单位
Chunk Size：每个 chunk 的大小，决定信息颗粒度

Chunk Size需要选择合适的大小，太大或者太小都有不好的地方。

Chunk Size太大
单个 chunk 会包含更多信息，上下文更完整，但也更容易混入无关内容。检索时可能因为局部内容匹配而召回一整段大文本，导致模型看到的信息变杂，回答更容易跑偏。同时，大 chunk 会占用更多上下文窗口，降低一次能放入的参考资料数量。

Chunk Size太小
单个 chunk 更容易精准命中具体信息，但上下文容易断裂。一个完整概念可能被拆散到多个 chunk 里，模型只拿到其中一部分时，就容易理解不完整。太小的 chunk 还会增加 chunk 总数，让索引、检索和排序的成本变高。

核心在于：**一个 chunk 应该能独立表达一个完整意思，同时尽量少包含无关信息。**
![Pasted image 20260608231828](/blog-assets/rag-guide-2-knowledge-base/Pasted%20image%2020260608231828.png)


**切分的策略**有多种，区别在于**按什么边界切，以及是否保留语义完整性**。

| 策略     | 适用场景                  | 特点                  |
| ------ | --------------------- | ------------------- |
| 固定长度切分 | 原始文本、格式混乱数据           | 简单稳定，按字符数或 token 数切 |
| 结构化切分  | Markdown、HTML、技术文档、博客 | 按标题、段落、小节等天然结构切     |
| 语义切分   | 长文章、知识库、问答材料          | 按话题变化或语义完整性切        |
| 滑窗重叠切分 | 上下文依赖强的内容             | 相邻 chunk 保留部分重复内容   |
![Pasted image 20260608231909](/blog-assets/rag-guide-2-knowledge-base/Pasted%20image%2020260608231909.png)
**固定长度切分**  
按固定字符数或 token 数切，比如每 500 token 切一块。实现简单，适合结构不清晰的文本，但可能把一句话、一个概念或一个表格从中间切断。

**结构化切分**  
按文档结构切，比如标题、段落、列表、Markdown 层级、HTML 标签。它更符合人写文档的方式，适合技术文档、产品文档、博客、说明书。

**语义切分**  
根据语义变化切，比如一个话题结束后再切到下一个 chunk。它更关注内容是否完整，适合长文章、知识库和解释型材料，但实现成本更高。

**滑窗重叠切分**  
在相邻 chunk 之间保留一段 overlap，比如前一个 chunk 的最后 100 token 出现在下一个 chunk 开头。它能缓解上下文断裂问题，适合概念连续、引用较多、上下文依赖强的内容。

推荐切分方法：**优先按结构切分，再用 chunk size 控制大小，必要时加 overlap。**


## 切片向量化

文档被切成一个个 chunk 后，还不能直接用于语义检索。因为计算机无法直接判断两段自然语言“意思是否相近”，所以需要先把每个 chunk 转换成向量。

这个过程叫 **Embedding**，也就是把文本映射成一组数字。可以把向量理解成文本在“语义空间”里的坐标：意思相近的文本，坐标距离更近；意思差异大的文本，坐标距离更远。

例如：
1.你中午吃的什么？ 
2.中午你吃的什么？
3.你晚上吃的什么？

第 1 句和第 2 句的词序不同，但表达的意思几乎一样，都是在问“中午吃了什么”。经过 Embedding 之后，它们在向量空间里的距离会更近。

第 3 句和前两句只有一个词不同：把“中午”换成了“晚上”。这个变化会改变问题的时间条件，所以它和前两句的语义距离会更远。

这就是向量化的作用：**它关注文本表达的意思，而不只是表面的字词是否相同。**
![Pasted image 20260608231936](/blog-assets/rag-guide-2-knowledge-base/Pasted%20image%2020260608231936.png)
在向量化时，每个 chunk 通常会被处理成三类信息：

|信息|作用|
|---|---|
|原文|后续提供给大模型阅读和回答|
|向量|用来计算语义相似度|
|元数据|记录来源、标题、章节、页码、编号等信息|

所以，切片向量化的本质是：**把文档内容变成可计算、可检索的语义表示。**

## 存入向量数据库

把切片向量化之后要将向量保存起来，保存这些向量的地方，就是向量数据库。

向量数据库和普通数据库的关注点不同。普通数据库更擅长精确查询，比如根据 ID、标题、时间、分类查数据；向量数据库更擅长相似度查询，比如“找出和这个问题语义最接近的几段内容”。

一条存入向量数据库的数据，通常包含三部分：

```json
{
  "text": "Chunk Size 太大会导致检索结果混入无关信息...",
  "vector": [0.12, -0.08, 0.45],
  "metadata": {
    "source": "rag-blog.md",
    "section": "文档切分",
    "chunk_index": 3
  }
}
```
这里的 vector 用来做相似度检索，text 是最终交给大模型参考的原文，metadata 用来记录这段内容从哪里来。
![Pasted image 20260608232009](/blog-assets/rag-guide-2-knowledge-base/Pasted%20image%2020260608232009.png)

当用户提问时，系统会把问题也转换成向量，然后去向量数据库里搜索距离最近的 chunk。返回的 Top K 个 chunk，就是最可能和问题相关的文档片段。
![Pasted image 20260608232025](/blog-assets/rag-guide-2-knowledge-base/Pasted%20image%2020260608232025.png)
这个过程可以理解成：

1. 文档提前切片
2. 每个切片提前向量化并存入数据库
3. 用户提问时，把问题也向量化
4. 在向量数据库中找语义最接近的内容
5. 把召回的内容交给大模型生成答案

**向量数据库保存的是“可被语义检索的知识索引”**
![Pasted image 20260608232043](/blog-assets/rag-guide-2-knowledge-base/Pasted%20image%2020260608232043.png)
